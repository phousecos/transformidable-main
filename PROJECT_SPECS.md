## 1. Project Overview

Transformidable CMS is a headless content management system built on **Payload CMS v3**. It serves as the central content hub for the Transformidable media ecosystem — a network of brand properties spanning personal branding, coaching, project management, interview prep, and membership content.

The CMS manages articles, podcast episodes, newsletters, authors, sponsors, and multi-brand syndication from a single admin interface. Each piece of content can be tagged with **Brand Pillars** and syndicated across any combination of five brand domains.

### Brand Properties

| Domain                      | Focus                        |
| --------------------------- | ---------------------------- |
| `jerribland.com`            | Jerri Bland personal brand   |
| `unlimitedpowerhouse.com`   | Coaching & business content  |
| `agentpmo.com`              | Project management content   |
| `prept.com`                 | Interview preparation        |
| `lumynr.com`                | Membership-exclusive content |

### Request Flow

1. **Admin users** access `/admin` — served by Payload's built-in admin UI
2. **API consumers** (brand sites) call `/api/<collection>` (REST) or `/api/graphql`
3. Payload enforces role-based access control on every request
4. Media uploads route through Vercel Blob Storage in production; local filesystem in dev
5. Database schema is auto-pushed via Drizzle on every build (no manual migrations)

## 4. Collections (Data Models)

### 4.1 Users

**Slug:** `users` | **Auth:** email + password

Administrative accounts for the CMS. Each user has a single role that governs what they can see and do.

| Field                | Type         | Notes                                  |
| -------------------- | ------------ | -------------------------------------- |
| `email`              | email (auth) | Unique, used for login                 |
| `firstName`          | text         | Optional                               |
| `lastName`           | text         | Optional                               |
| `role`               | select       | `admin` · `editor` · `brandContributor` · `sponsorManager` |
| `assignedBrandPillar`| relationship | → Brand Pillars. Shown only for Brand Contributors |

**Access:** Only admins can create/delete users. Users can update their own profile (but not their role).

---

### 4.2 Media

**Slug:** `media` | **Upload collection**

Handles image, audio, and PDF uploads with automatic image resizing.

| Field     | Type   | Notes                          |
| --------- | ------ | ------------------------------ |
| `alt`     | text   | Required — accessibility text  |
| `caption` | text   | Optional                       |

**Accepted MIME types:** `image/*`, `audio/*`, `application/pdf`

**Auto-generated image sizes:**

| Name        | Dimensions   | Use Case       |
| ----------- | ------------ | -------------- |

### 4.3 Articles

**Slug:** `articles` | **Versions:** drafts enabled

Primary editorial content with multi-brand syndication.

| Field            | Type         | Notes                                        |
| ---------------- | ------------ | -------------------------------------------- |
| `title`          | text         | Required                                     |
| `slug`           | text         | Unique, auto-generated from title            |
| `body`           | richText     | Required — Lexical editor                    |
| `excerpt`        | textarea     | 2-3 sentence summary for cards               |
| `author`         | relationship | → Authors (required)                         |
| `publishDate`    | date         | Day + time picker                            |
| `featuredImage`  | upload       | → Media                                      |
| `brandPillars`   | relationship | → Brand Pillars (many)                       |

| `status`         | select       | `draft` → `review` → `scheduled` → `published` |

## 5. Role-Based Access Control

The system uses four roles with hierarchical permissions:

| Capability                    | Admin | Editor | Brand Contributor | Sponsor Manager |
| ----------------------------- | :---: | :----: | :---------------: | :-------------: |
| Manage users                  |  ✓    |        |                   |                 |
| Manage brand pillars          |  ✓    |        |                   |                 |
| Delete any content            |  ✓    |        |                   |                 |
| Full content CRUD             |  ✓    |   ✓    |                   |                 |
| Change article/episode status |  ✓    |   ✓    |                   |                 |
| Create articles & media       |  ✓    |   ✓    |        ✓          |                 |
| Edit own-pillar articles      |  ✓    |   ✓    |        ✓          |                 |
| Manage sponsors               |  ✓    |        |                   |        ✓        |
| Read own profile              |  ✓    |   ✓    |        ✓          |        ✓        |

Access control logic is centralized in `src/access/checkRole.ts`.

## 6. API Surface

Payload auto-generates both REST and GraphQL endpoints for every collection.

### REST API

Base path: `/api`

| Method | Endpoint                    | Description              |
| ------ | --------------------------- | ------------------------ |
| GET    | `/api/articles`             | List / query articles    |
| GET    | `/api/articles/:id`         | Get single article       |
| POST   | `/api/articles`             | Create article           |
| PATCH  | `/api/articles/:id`         | Update article           |
| DELETE | `/api/articles/:id`         | Delete article           |

The same pattern applies to: `users`, `media`, `authors`, `podcast-episodes`, `brand-pillars`, `sponsors`, `newsletter-issues`.

**Authentication:** Pass a Bearer token (obtained via `POST /api/users/login`) in the `Authorization` header.

### GraphQL

Endpoint: `/api/graphql`

Payload generates queries and mutations for all collections. Introspection is available for schema discovery.

### Diagnostic Endpoints

| Endpoint      | Description                                           |
| ------------- | ----------------------------------------------------- |
| `/health`     | Health check — Sharp, Payload init, DB connectivity   |
| `/api/diag`   | Detailed diagnostics — user count, collections, auth  |

## 7. Content Workflow

Content (articles, podcast episodes) follows a four-stage publishing pipeline:

```
Draft  →  Review  →  Scheduled  →  Published
```

- **Draft:** Initial creation. Visible only to authenticated users with access.
- **Review:** Ready for editorial review. No automated behavior — status marker only.
- **Scheduled:** Approved and awaiting publish date. (Note: automated publishing is not yet implemented — must be manually moved to Published.)
- **Published:** Live and publicly accessible via API.

Newsletter issues use a simplified three-stage pipeline: `Draft → Scheduled → Sent`.

Status transitions are restricted to admins and editors via field-level access control.

---

## 8. Multi-Brand Syndication

The syndication model works through two complementary mechanisms:

1. **Brand Pillars** — relationship tags that categorize content by brand focus area. Each pillar maps to a specific domain. Used for organizing content and scoping contributor access.

2. **syndicateTo** — explicit multi-select field on articles and episodes. Controls which brand domains are authorized to republish the content.

Consumer sites query the API filtering by their relevant brand pillar and/or syndication target to retrieve only the content intended for their domain.

## 11. Build & Deployment

### Scripts

| Script                  | Command                                                        |
| ----------------------- | -------------------------------------------------------------- |
| `npm run dev`           | Start Next.js dev server with hot reload                       |
| `npm run build`         | Run schema push → generate import map → build Next.js          |
| `npm run migrate`       | Push database schema only                                      |
| `npm start`             | Start production server                                        |
| `npm run lint`          | Run ESLint                                                     |
| `npm run generate:types`| Regenerate `payload-types.ts` from collection configs          |

### Build Pipeline

```
npm run build
  ├── tsx src/migrate.ts              # Push DB schema via Drizzle
  ├── npx payload generate:importmap  # Generate admin UI import map
  └── next build                      # Compile Next.js application
```

### Database Strategy

The project uses **schema push** (`push: true` in the Postgres adapter config) rather than migration files. On every build, the Drizzle adapter compares the current collection configs against the database schema and applies changes automatically. This simplifies development but means schema changes are applied directly — there is no rollback mechanism.

### Deployment

- **Platform:** Vercel
- **Branch deploys:** Automatic on push
- **SSL:** Auto-configured. On Vercel, the connection string is patched to use `sslmode=no-verify` for Neon compatibility.

## 12. Known Limitations & Future Work

### Current Limitations

- **No automated scheduled publishing.** The `scheduled` status is a manual marker — there is no cron job or serverless function that automatically transitions content to `published` at the scheduled time.
- **No test suite.** No unit, integration, or E2E tests exist. The project has no testing framework configured.
- **No migration rollbacks.** Schema push mode means changes are applied forward-only. If a schema change breaks something, it must be fixed manually.
- **Audio hosting is external.** Podcast episodes store an `audioUrl` string rather than uploading audio directly. The actual audio hosting is managed outside this system.
- **No frontend rendering.** The CMS is purely an admin + API backend. Brand sites consume the API but are separate projects.

### Recommended Next Steps

- **Add a testing framework** (Vitest or Jest) with collection-level integration tests
- **Implement scheduled publishing** via Vercel Cron or a similar mechanism
- **Add webhook notifications** for content status changes (useful for triggering builds on consumer sites)
- **Implement API rate limiting** for public-facing endpoints
- **Add content preview URLs** so editors can preview articles on brand sites before publishing
- **Consider migration files** for production schema changes once the schema stabilizes
- **Add slug uniqueness validation** with user-friendly error messages
- **Implement content versioning UI** — drafts are enabled but version comparison isn't customized
